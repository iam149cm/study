### ✅ 간단 개념 학습
#### ⭐ ECID (Experience Cloud ID)
- ref link : https://experienceleague.adobe.com/ko/docs/id-service/using/intro/overview

#### ☝🏻 정의 
- ECID는 Adobe Experience Cloud의 모든 솔루션(Analytics, Target, Audience Manager 등)에서 사용자를 동일하게 식별하기 위한 고유 ID입니다. 
- 과거에는 `s_vi` 쿠키나 AppMeasurement의 `visitorID`를 사용했지만, 이제는 ECID 서비스가 표준입니다.

#### ✌🏻 실무 포인트
- 크로스도메인 추적: 서로 다른 도메인 간 방문자 연동이 필요할 때 필수
- 세션 중복: ECID 설정 오류 시, 한 명의 방문자가 여러 명으로 중복 집계되는 문제 발생
- 디버깅: `s_ecid` 쿠키와 네트워크 요청의 `mid` 값이 일치하는지 확인 필요
- 시험 팁 : 방문자 식별 관련 문제가 나오면, 정답의 80%는 ECID와 관련된 개념에 집중됩니다.

#### ⭐ Web SDK vs AppMeasurement
- ref link : https://experienceleague.adobe.com/ko/docs/analytics/implementation/prepare/comparison
- Web SDK는 최신 데이터 수집 방식으로, 통합 태깅과 실시간 데이터 전송이 강점
- 마이그레이션 시 consent 설정, 데이터 맵핑, 서버 설정을 철저히 점검해야 함

#### ☝🏻 Web SDK (Alloy.js)
- Adobe의 최신 데이터 수집 라이브러리로, Analytics·Target·Audience Manager 등의 태깅을 통합 지원합니다.
- 데이터를 Edge Network로 전송해 실시간 처리 및 Audience Activation이 용이합니다.

#### ✌🏻 AppMeasurement.js
- 기존 방식의 데이터 수집 스크립트로, 현재도 많이 사용되지만 Web SDK로 점차 전환 중입니다.

#### 👉🏻 마이그레이션 시 체크리스트
- Consent 설정 확인 (GDPR/CCPA 대응 필수)
- 데이터 맵핑 표준화 (XDM 스키마 설계)
- ECID 서비스 통합 여부 확인
- 서버 전송 설정(Server-Side Forwarding) 검토


### 🎯 마지막으로 간단한 예상문제를 하나 풀어볼까요? 🤗
#### Q1. After migrating to Web SDK, the number of Unique Visitors suddenly doubled. Which of the following is the most likely cause?
    A. Incorrect ECID configuration
    B. Wrong currency setting in the report suite
    C. Expired prop variable mapping
    D. Using an outdated Debugger extension

- 정답 : A. Incorrect ECID configuration 
- 해설 : ECID가 정상적으로 설정되지 않으면 동일 사용자가 여러 방문자로 중복 집계됩니다. 마이그레이션 시 ECID 통합 여부와 쿠키 도메인 설정을 반드시 점검해야 합니다.
